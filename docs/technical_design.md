# 기술 설계 문서 (Technical Design Document)

## 1. 아키텍처 구조 (Architecture)
게임 클라이언트(`client/`)가 커지면서 스파게티 코드가 되는 것을 방지하기 위해 **상태 머신(State Machine)** 패턴을 적용합니다.

*   **`main.lua`**: 진입점. 전역 상태를 관리하고 각 상태(State) 파일로 `update`와 `draw`를 위임합니다.
*   **상태 파일 분리**:
    *   `state_explore.lua`: 1인칭 던전 탐색 및 이동 로직 담당.
    *   `state_combat.lua`: 턴제 전투 로직, 커맨드 입력 및 체력 처리 담당.
    *   `state_hub.lua`: 거점 화면 (안전 가옥) 및 정비 로직 담당.

## 2. 데이터 관리 (Data Models)
하드코딩을 피하고 확장을 쉽게 하기 위해 모든 데이터를 분리된 Lua 테이블(모듈)로 관리합니다.

*   **`data_party.lua`**: 5인 파티의 이름, 클래스, 현재 HP/SP, 스탯, 장비 슬롯을 관리합니다.
*   **`data_enemy.lua`**: 적의 능력치(HP, 공격력, 속도), 드랍 아이템, 그리고 **생성할 아트 모드(mode)와 프롬프트**를 함께 정의하여 필요 시 즉석에서 파이프라인을 호출할 수 있게 합니다.

## 3. 던전 렌더링 방식 (Pseudo-3D)
진짜 3D 엔진을 쓰지 않고 DRPG 특유의 1인칭 시점을 구현하는 고전적인 방식을 사용합니다.

1.  **배경 레이어**: AI가 생성한 `[MAP]` 모드의 이미지 (예: 네온 복도)를 화면 전체에 렌더링.
2.  **적 레이어**: 해당 칸에 적이 있다면 화면 중앙에 `[MONSTER]` 모드의 이미지를 오버레이.
3.  **UI 레이어**: 미니맵, 파티 상태창, 커맨드 메뉴를 맨 위에 렌더링.

## 4. 이동 및 맵 좌표계 (Grid System)
*   맵은 2차원 배열(Grid)로 구성되며 `0`은 벽, `1`은 길, `2`는 문, `3`은 이벤트(적/아이템)로 정의합니다.
*   플레이어는 `(x, y)` 좌표와 `방향(North, East, South, West)` 값을 가집니다.
*   방향키 [위]를 누르면 플레이어가 바라보는 방향으로 1칸 이동을 시도하며, 벽이 아닐 경우 좌표가 변경되고 새로운 맵 배경 이미지가 필요하다면 갱신합니다.

## 5. 단계별 개발 계획 (Next Steps in Lua)
1.  **모듈화 리팩토링**: 현재 `main.lua`에 섞여 있는 전투 UI를 `state_combat.lua`로 분리.
2.  **State 전환 시스템**: `[Space]` 키를 눌러 "탐험 상태"와 "전투 상태"를 오가는 뼈대 작성.
3.  **그리드 맵 프로토타입**: 5x5 사이즈의 하드코딩된 맵 배열을 만들고, 미니맵 상에서 좌표가 이동하는 기능 구현.
